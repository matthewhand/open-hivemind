## 2026-03-11 - SSRF Prevention in LLM Packages
**Vulnerability:** Multiple LLM provider packages (e.g., openwebui, flowise, openswarm, openai) were vulnerable to Server-Side Request Forgery (SSRF) because they used configured or dynamically generated URLs for `axios` requests without validating if those URLs pointed to private or reserved IP ranges.
**Learning:** Security utilities like `isSafeUrl` must be placed in globally accessible workspace packages (e.g., `@hivemind/shared-types` or a dedicated `utils` package) rather than main `src/` directories so that all child packages can access them without creating dependency cycles or relative import nightmares.
**Prevention:** Always wrap dynamically generated external HTTP requests (via axios, fetch) with `isSafeUrl` validation checks. Enforce this via a custom ESLint rule if possible.

## 2026-03-13 - Arbitrary Command Execution via MCP Stdio Transport
**Vulnerability:** The MCP (Model Context Protocol) integration in `src/server/routes/mcp.ts` allowed an administrator to add MCP servers via a `stdio://` URL scheme. The path after `stdio://` was passed directly as the command to Node.js's `spawn` (via `@modelcontextprotocol/sdk/client/stdio.js`) without any validation. An attacker with admin privileges (or via an SSRF/CSRF, though admin was required) could execute arbitrary local commands on the server by supplying URLs like `stdio:///bin/sh` or `stdio://node`.
**Learning:** Even when a URL scheme (like `stdio://`) implies local execution rather than a network request, the resulting command string must be treated as untrusted user input and subjected to strict command injection / executable allowlist validation.
**Prevention:** Always validate executable paths and commands derived from user input against a strict allowlist (or robust blocklist/pattern list). We extracted the command validation logic from `MCPProviderManager` into a reusable `validateMCPCommand` helper and applied it to the `stdio://` parser. We also added language interpreters (node, python, perl, ruby) to the blocklist since they can execute arbitrary inline code via their arguments (e.g., `node -e "..."`).
