## 2025-02-27 - Fix SSRF via HTTP Redirects Bypass
**Vulnerability:** The application was vulnerable to SSRF through HTTP redirects. The `isSafeUrl` function correctly checked the initial URL, but the underlying generic HTTP client (`httpClient.ts`) used `fetch` with the default `redirect: 'follow'` behavior. An attacker could bypass the `isSafeUrl` guard by supplying a safe, public URL that responded with a 3xx redirect to a private or internal IP (e.g., `169.254.169.254` or `127.0.0.1`).
**Learning:** Checking the URL against an SSRF guard before initiating an HTTP request is insufficient if the underlying HTTP client blindly follows redirects. The target of the redirect must also be validated, or automatic redirects must be disabled to prevent this class of SSRF bypass.
**Prevention:** Always explicitly configure HTTP clients (e.g., `fetch` with `redirect: 'manual'`, or `axios` with `maxRedirects: 0`) when validating URLs for SSRF. If redirects must be followed, manually inspect the `Location` header and run it through the SSRF guard before proceeding.
